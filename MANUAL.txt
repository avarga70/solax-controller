================================================================================
  SOLAR CONTROLLER — USAGE MANUAL
  GoodWe 10K ET  |  HA Daemon  |  Czech OTE Spot Prices
================================================================================

OVERVIEW
--------
solar_controller.py is a daemon that runs every few minutes (default 5) and
optimises your GoodWe 10K ET inverter based on:

  • Czech spot electricity prices  (from local MySQL table EL_dailypr, CZK/MWh)
  • Current battery SOC and inverter state  (via goodwe UDP library)
  • Weather-based PV generation forecast  (forecast.solar, free, no account)
    Falls back to historical PV_run patterns if the API is unavailable.
  • Planned grid outage schedule  (planned_outages.json)

It never exports battery energy to the grid.  Instead it controls:
  1. Inverter operation mode  (GENERAL / ECO_CHARGE / BACKUP)
  2. Battery minimum discharge SOC  (written via write_setting)

TEST MODE runs the full decision logic and logs everything but sends NO commands
to the inverter.  Safe to leave running for days before going live.


FILES
-----
  solar_controller.py     Main daemon
  check_inverter.py       One-shot diagnostic — lists all inverter settings
  config.env              Runtime configuration  (copy from config.env.example)
  planned_outages.json    Scheduled grid outage windows
  solar.log               Rotating log  (5 MB × 7 backups)
  requirements.txt        Python dependencies


QUICK START
-----------
1.  Install dependencies (once):
      python3 -m venv .venv
      .venv/bin/pip install -r requirements.txt

2.  Configure:
      cp config.env.example config.env
      # Edit config.env — set INVERTER_IP, MYSQL_*, NODE_NAME, TEST_MODE=1

3.  Test a single cycle:
      .venv/bin/python3 solar_controller.py --test --once

4.  Run as a service (systemd):
      sudo systemctl start  solar-controller
      sudo systemctl enable solar-controller   # start on boot
      sudo systemctl status solar-controller

5.  Go live:
      # In config.env on each node:  TEST_MODE=0
      sudo systemctl restart solar-controller


CONFIGURATION  (config.env)
----------------------------
Required:
  INVERTER_IP            IP address of the GoodWe inverter (UDP port 8899)
  MYSQL_HOST             MySQL / Percona cluster VIP or hostname
  MYSQL_DB               Database name  (e.g. GRAFANA)
  MYSQL_USER             Database user
  MYSQL_PASSWORD         Database password

Mode:
  TEST_MODE              1 = dry-run (default), 0 = live control
  NODE_NAME              Unique name for this node in HA mode  (e.g. node1)
  LOOP_INTERVAL_MINUTES  Cycle interval in minutes              (default 5)

Battery:
  BATTERY_CAPACITY_WH    Usable battery capacity in Wh             (default 7100)
  CHARGE_RATE_W          Max charge power in W                     (default 2750)
  MIN_SOC                Normal daytime minimum SOC %              (default 20)
  NIGHT_MIN_SOC          Minimum SOC % at night (21:00–06:00)      (default 20)
  MIN_SOC_FLOOR          Absolute minimum ever written to inverter  (default 20)
  MIN_SOC_SAVE           Min SOC raised during cheap-price charge   (default 50)
  CHARGE_SOC_LIMIT       Maximum charge target %                   (default 98)
  ECO_MIN_SOC            Min SOC during ECO_CHARGE (PV buffer)     (default 95)

Price thresholds (dynamic, computed from next ~24 h of prices):
  CHEAP_PERCENTILE       Bottom N% of next-24h prices = cheap      (default 25)
  EXPENSIVE_PERCENTILE   Top N% of next-24h prices = expensive      (default 75)
  PRICE_CHEAP            Fallback cheap threshold CZK/MWh           (default 1500)
  PRICE_EXPENSIVE        Fallback expensive threshold CZK/MWh       (default 3000)
  RECHARGE_HORIZON_H     Tomorrow hours included in threshold window (default 13)
                         Prevents high afternoon peaks from masking today's cheap
                         overnight recharge opportunities.
  PRICE_NO_EXPORT        Below this price, absorb PV into battery   (default 200)
                         rather than exporting at near-zero revenue.

Arbitrage profitability gate:
  DEVIATION_FEE          Czech deviation fee per kWh  CZK/MWh      (default 390)
                         Added to buy price when computing break-even sell price.
  BATTERY_EFFICIENCY     Round-trip charge/discharge efficiency 0–1  (default 0.92)

Distribution tariff surcharge (ČEZ Distribuce 2025):
  DIST_FEE_NORMAL        Normal-hour distribution fee  CZK/MWh     (default 140.97)
  DIST_FEE_PEAK          Peak-hour distribution fee    CZK/MWh     (default 913.27)
  DIST_PEAK_HOURS        Comma-separated peak hours 0–23           (default 8,12,15,19)
                         During these hours the effective buy price is increased by
                         the delta (DIST_FEE_PEAK − DIST_FEE_NORMAL) before the
                         cheap-price and break-even checks.

Smart discharge:
  SMART_DISCHARGE        1 = optimise pre-peak discharge (default),  0 = conservative
                         When enabled:
                         • discharge_start is selected by ranking hours on
                           effective_price × expected_load, filtered to hours where
                           discharging beats the cheapest upcoming recharge
                           (round-trip break-even check).
                         • The Rule-6 overnight hold floor is lowered from
                           CHARGE_SOC_LIMIT to the minimum SOC actually needed for
                           the morning load window, so the battery can serve house
                           load at profitable pre-peak prices instead of sitting
                           idle at 98%.
                         Set to 0 to revert to the original conservative behaviour
                         if the new strategy produces unexpected results.

Scheduled fixed loads (DHW water heater):
  DHW_START_HOUR         Hour hot-water heater switches on (inclusive)  (default 14)
  DHW_END_HOUR           Hour hot-water heater switches off (exclusive) (default 15)
  DHW_WH                 Wh guaranteed for water heater regardless of   (default 0)
                         historical averages.  Set to actual heater Wh (e.g. 2550).

Prep-charge timing:
  PREP_CHARGE_RATE_KW    Assumed grid charge rate in kW  (default 3.0)
                         Used to compute how many hours before the morning peak the
                         overnight prep-charge (Rule 2.5) should start.

Planned outages:
  OUTAGE_PRE_CHARGE_SOC  Target SOC % to reach before outage       (default 98)
  OUTAGE_LEAD_HOURS      Maximum pre-charge window in hours         (default 4)
                         Actual lead time is calculated dynamically from current
                         SOC and CHARGE_RATE_W.

HA (high-availability, two-node):
  HA_ENABLED             1 = enable leader election via MySQL lease  (default 0)

History / patterns:
  HISTORY_MONTHS         Months of PV_run history used for patterns (default 24)
  PV_SOURCE              psource filter value in PV_run table       (default VV_PV)

PV generation forecast (forecast.solar — free public API, no account needed):
  PV_LAT                 Site latitude  (decimal degrees, e.g. 49.8239)
  PV_LON                 Site longitude (decimal degrees, e.g. 18.1256)
  PV_DEC                 Panel tilt angle °  0=horizontal, 30=typical roof
  PV_AZ                  Panel azimuth °  -90=East, 0=South, 90=West
  PV_KWP                 Installed peak power in kWp  (e.g. 4.56)
                         Set PV_KWP=0 to disable and use historical patterns only.
                         The API is called once per hour and cached; rate limit is
                         12 req/hour — well within the 6 req/hour the daemon uses.


DECISION RULES  (priority order)
---------------------------------
Rules are evaluated once per cycle.  The first matching rule wins.

  0. PLANNED OUTAGE
     • Inside outage window   → GENERAL, min SOC = OUTAGE_PRE_CHARGE_SOC
     • Outage approaching     → ECO_CHARGE, min SOC = OUTAGE_PRE_CHARGE_SOC
       (lead time = Wh to charge ÷ CHARGE_RATE_W, max OUTAGE_LEAD_HOURS)

  1. EMERGENCY SOC
     • Battery at/below MIN_SOC_FLOOR → ECO_CHARGE (fast grid charge)

  1.5 NEGATIVE PRICE
     • Spot price ≤ 0  → ECO_CHARGE  (absorb free electricity into battery)

  2. CHEAP PRICE  (current price ≤ cheap threshold)
     • Battery not full AND arbitrage profitable AND PV won't fill it
       → ECO_CHARGE, min SOC = CHARGE_SOC_LIMIT
     Arbitrage gate: grid charging is skipped if the battery cannot earn back
     the round-trip cost at the best upcoming sell price:
       min_sell_price = (buy_price + DEVIATION_FEE) / BATTERY_EFFICIENCY

  2.5 NIGHT PREP-CHARGE  (overnight only)
     • Expected load before the next cheap window exceeds battery headroom
       → ECO_CHARGE at the cheapest available overnight hour.
     Ensures the battery arrives at the morning peak with enough charge.

     Oscillation guard: skipped if Rule 3 discharge is already profitable now
     (buying to 98% only to immediately discharge would waste round-trip losses).

     Opportunity-cost gate: skipped when deeply negative midday prices are
     coming — preserving battery headroom to absorb free electricity is worth
     more than the overnight pre-charge saving.

  3. EXPENSIVE NOW  (current price ≥ expensive threshold)
     • Hour ≥ discharge_start:
       → GENERAL, min SOC = MIN_SOC_FLOOR  (battery covers house load).
       Floor raised to preserve_min_soc if expensive morning peak hours arrive
       before the next cheap window; lowered back when a cheaper overnight
       pre-charge is coming (_precharge_possible) — discharge freely now.
     • Hour < discharge_start AND a cheap recharge window comes before
       discharge_start:
       → GENERAL, discharge now — battery will recharge and discharge again
     • Continuous expensive block (no cheap gap until discharge_start):
       → GENERAL, discharge through the whole block

  4–5. EXPENSIVE HOURS COMING / END-OF-DAY
     Rule 4: Expensive hours coming later today with PV running — raise floor
     to preserve SOC for the upcoming peak.

     Rule 5: End-of-day hold/discharge window (hour ≥ eod_hour, not yet night).
     eod_hour is computed dynamically each cycle as the last hour with > 500 Wh
     of forecast PV output + 1 (EOD_HOUR config is a fallback only).
     • Before discharge_start → GENERAL, min SOC = max(ECO_MIN_SOC, current SOC)
       (holds at current level — prevents Rule 2 from topping up only for
       Rule 5 to walk it back down, causing charge/discharge oscillation)
     • At/after discharge_start → GENERAL, overnight floor by morning outlook.

  6. OVERNIGHT HOLD / DRAIN  (is_night, expensive before next cheap window)
     • drain_start = latest hour h where expected_load(h → next_cheap) fits
       within the battery's usable range.
     • hour < drain_start (hold phase):
       → GENERAL, min SOC = hold_floor  (grid covers load until drain_start)
       With SMART_DISCHARGE: hold_floor is lowered from CHARGE_SOC_LIMIT to
       the minimum SOC needed at drain_start, so the battery can discharge
       profitably at pre-peak prices rather than sitting idle at 98%.
     • hour ≥ drain_start (drain phase):
       → GENERAL, min SOC = NIGHT_MIN_SOC  (battery drains to cover morning peak)

  DEFAULT
     → GENERAL, min SOC = preserve_min_soc  (preserves charge for morning peak)

discharge_start is the earliest hour the battery should freely discharge.
With SMART_DISCHARGE it is the beginning of the contiguous block of hours
where discharging beats buying at the cheapest upcoming recharge price,
ranked by effective_price × expected_load.  Without SMART_DISCHARGE it is
the hour with the highest raw price that the battery can cover.


PLANNED OUTAGES  (planned_outages.json)
----------------------------------------
Edit the file (no restart needed — read every cycle):

  [
    {
      "start": "2026-03-05 08:00",
      "end":   "2026-03-05 14:00",
      "note":  "planned grid maintenance"
    }
  ]

Multiple entries are supported.  Past entries are ignored automatically.
Lead time is calculated per cycle from current SOC — no manual adjustment needed.


MONITORING
----------
Tail the live log:
  tail -f /home/avarga/AI/solar/solar.log

Query decision history (MySQL):
  SELECT logged_at, price_now, current_mode, target_mode, target_min_soc, reason
  FROM solar_decisions
  ORDER BY logged_at DESC
  LIMIT 20;

Service status:
  sudo systemctl status solar-controller

Inverter diagnostics (lists all settings keys — find the SOC limit key):
  .venv/bin/python3 check_inverter.py 192.168.0.148


HIGH-AVAILABILITY (two nodes)
------------------------------
Both nodes run the daemon.  Only the LEADER connects to the inverter and applies
commands.  Leader election uses a MySQL lease table (solar_leader_lease).

  config.env on node1:  HA_ENABLED=1  NODE_NAME=node1
  config.env on node2:  HA_ENABLED=1  NODE_NAME=node2

If the leader node fails, the standby takes over within 2 × LOOP_INTERVAL_MINUTES + 30 s
(~40 s with the default 5-minute cycle).  The inverter's last written settings remain active until
the new leader takes a cycle.

Deploy changes to both nodes:
  ./deploy.sh --restart
